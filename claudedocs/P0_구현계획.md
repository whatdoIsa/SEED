## 마일스톤 개요와 순서 근거

```
M0 기반  →  M1 엔진 확장  →  M2 시장 UI  →  M3 학습 시스템  →  M4 복기·시즌  →  M5 온보딩·계측·마감
(스키마)     (배속·체결·시나리오)  (토스 레지스터)   (레슨·튜토리얼)      (리포트·부검)       (묶기·검증)
```

- **엔진(M1)이 UI(M2)보다 먼저**: 배속·체결 내역·시나리오는 화면들이 소비하는 원료다. UI부터 만들면 목데이터로 두 번 일하게 된다.
- **학습(M3)은 시장 UI(M2) 위에**: 슬리피지 튜토리얼·미션은 시장 화면 안에서 벌어진다.
- **복기(M4)는 태그 데이터(M2의 태그 시트) 이후**: L1 복기는 `TradeLog` 집계이므로 데이터가 쌓이는 구조가 먼저다.
- **온보딩(M5)은 마지막**: 온보딩은 "완성된 첫 5분"의 편집본이다. Lv0 화면·튜토리얼·레슨이 있어야 이어붙일 수 있다.
- **계측은 M0에서 파사드만 만들고 각 티켓이 이벤트를 심는다** — 마지막에 몰아 넣으면 누락된다.

## 크리티컬 패스와 병렬화

- **크리티컬 패스**: M0-2 → M1-2 → M2-3 → M3-3(슬리피지 튜토리얼) — 제품의 존재 증명이 이 사슬 끝에 있다.
- **병렬 가능**: M1-3/M1-4(시나리오)는 M2와 독립적으로 진행 가능. M0-3(토큰)·M0-4(계측)는 언제든 선행 가능.
- **가장 큰 리스크 티켓**: M1-3(ScenarioPreset — 결정론적 재현)과 M3-1(레슨 엔진 — 선언적 스키마). 이 둘은 설계를 먼저 굳히고 착수할 것.
- **범위 방어선**: 일정이 밀리면 M2-2 호가창을 "4호가 읽기 전용"으로 축소(주문은 M2-3 모달로 충분), M4-2 공유 이미지 내보내기를 이연. **슬리피지 튜토리얼과 태그 시트는 절대 자르지 않는다** — 차별화의 최소 단위.
